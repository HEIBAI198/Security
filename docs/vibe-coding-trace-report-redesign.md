# SupplyGuard KG 溯源报告改造说明

## 目标

当前溯源报告已经能展示风险评分、攻击路径、证据矩阵和可信断点，但对普通用户来说还偏“技术结果展示”。为了更贴合 Vibe Coding 安全场景，报告需要从“列出扫描结果”升级为“告诉用户发生了什么、为什么危险、先处理什么、还缺什么证据”。

推荐把报告改成以下定位：

```text
面向 AI 编程 / Vibe Coding 场景的软件供应链攻击溯源报告

回答五个问题：
1. 这个风险可能是不是 AI 生成代码、AI 推荐依赖或 AI 生成配置引入的？
2. 风险从哪里进入项目？
3. 有没有进入构建链、产物或运行期？
4. 是否可能影响用户数据、凭据、发布渠道或下游系统？
5. 用户现在应该先做什么？
```

## 报告首屏应该先给结论

用户打开报告后，不应该先看到大量图表，而应该先看到一段可以直接理解的研判结论。

建议首屏增加“研判助手摘要”模块：

```text
建议优先处理「测试」项目中的最高风险链路：
证据可串成“供应链投毒到运行期异常”的攻击路径。

当前依赖侧首要对象为 axios，风险分 100；
运行期证据指向 app -> 45.83.64.12。

处置建议：
1. 隔离高危依赖。
2. 使用干净 runner 重新构建。
3. 校验产物哈希和 provenance。
4. 排查运行期异常外联。
5. 必要时吊销相关 Token 并回滚发布版本。
```

这个模块的作用是让用户不用先读完整报告，也能知道：

- 当前最危险的对象是谁。
- 攻击路径是否已经从依赖走到运行期。
- 为什么要优先处理。
- 立即应该做什么。

前端展示建议：

```text
[研判助手]
一句话结论：可能存在从依赖污染到运行期异常外联的供应链攻击路径。

首要风险对象：axios
风险分：100
运行期证据：app -> 45.83.64.12
处置优先级：立即隔离依赖，重新构建并校验产物。
```

## 增加扫描对象与证据范围

报告需要告诉用户：“系统到底看了哪些文件，哪些证据没有看见”。否则用户会误以为系统已经覆盖了所有风险。

建议增加“扫描范围”模块：

```text
## 一、扫描范围

项目名称：测试
文件总数：29
可扫描文件：23
依赖清单：3 个
CI/CD 配置：0 个
主要语言：JSON
```

建议继续展示文件类型覆盖：

| 文件类型 | 文件数量 | 扫描状态 | 说明 |
| --- | ---: | --- | --- |
| JSON | 10 | 已扫描 | 主要用于依赖清单、结果文件和结构化日志 |
| Markdown | 3 | 已识别 | 可作为说明文档或报告来源 |
| YAML | 2 | 已识别 | 未发现有效 CI/CD 配置 |
| 二进制 / 压缩包 | 1 | 已跳过 | 需要作为产物可信材料单独校验 |

注意：这里的数量不要写死，应该由扫描结果动态生成。截图里的 `文件总数 29`、`可扫描文件 23`、`依赖清单 3 个` 可以作为演示样例。

## 增加依赖清单与 CI/CD 配置识别结果

用户需要知道系统扫描的是哪些依赖文件，而不是只看到“供应链风险”四个字。

建议报告中加入：

```text
## 二、工程证据清单

### 已识别依赖清单

- sample-repo/package-lock.json
- sample-repo/package.json
- sample-repo/requirements.txt

### 已识别 CI/CD 配置

未发现 CI/CD 配置文件。
```

如果 CI/CD 配置缺失，报告要明确写成“证据缺口”，而不是只显示 0。

推荐文案：

```text
未发现 CI/CD 配置文件，因此本次无法完整判断：
- workflow 是否使用未固定版本的第三方 Action；
- workflow 权限是否过宽；
- 构建 runner 是否可信；
- 构建过程是否存在 curl | bash、远程脚本执行等风险。
```

这样用户会知道：系统不是没能力，而是当前材料不足。

## 增加证据完整性提醒

截图中有一条提示：

```text
No CI/CD configuration file was found.; 1 binary or archive files were skipped.
```

报告里不要直接保留英文原文，建议翻译成用户能看懂的中文：

```text
## 三、证据完整性提醒

本次扫描存在以下证据缺口：

1. 未发现 CI/CD 配置文件。
   影响：无法确认构建链中是否存在高权限 workflow、未固定 Action 或危险脚本。

2. 发现 1 个二进制或压缩文件被常规代码扫描跳过。
   影响：该文件不能按源码方式审计，需要进入“产物可信门禁”进行哈希、签名、provenance 和 attestation 校验。
```

建议把这部分放在报告前半部分，因为它直接影响用户对结论可信度的理解。

## 增加 GraphRAG + GNN 智能证据模块

你截图里的“GraphRAG + GNN 智能证据”很适合放进报告，而且它可以成为项目的创新点展示。

建议报告中增加：

```text
## 四、GraphRAG + GNN 智能证据

系统将依赖、代码、CI/CD、产物、日志和告警构建为供应链知识图谱，并使用 GraphRAG 召回相关证据，使用 GNN 对依赖节点进行风险评分。

### 图谱规模

- 图谱节点：199
- 图谱边：176
- 攻击路径：3
- GNN 覆盖：164 / 164
- 当前模型：numpy_graphsage_mean_aggregator

### 高风险依赖示例

| 依赖 | GNN 风险分 | 说明 |
| --- | ---: | --- |
| axios | 100% | 当前攻击路径中的首要依赖风险对象 |
| pyyaml | 47% | 存在中等图结构风险，需要结合漏洞、版本和调用关系继续判断 |
```

这里要注意：

- `GNN 47%` 不是“这个包一定恶意”的意思。
- 它表示这个依赖节点在当前供应链图谱中的风险排序信号。
- 最终结论要结合漏洞、脚本、版本、CI/CD、产物和日志证据一起判断。

推荐解释文案：

```text
GNN 风险分用于辅助排序，不单独作为定罪依据。
GraphRAG 负责从知识图谱中召回与当前问题相关的依赖、构建、产物和运行期证据。
当 GNN 高分依赖同时出现在攻击路径、构建日志或运行期异常中时，系统会提高该路径的处置优先级。
```

## 增加 Vibe Coding 风险归因

为了贴合你的比赛故事，报告里必须回答“这个风险是否可能和 Vibe Coding 有关”。

建议增加：

```text
## 五、Vibe Coding 风险归因

本次风险可能与 Vibe Coding 引入方式有关：

| 风险来源 | 是否命中 | 证据 |
| --- | --- | --- |
| AI 推荐依赖 | 疑似 | 项目中出现高风险依赖，且依赖进入 package-lock.json |
| AI 生成代码 | 未确认 | 当前未发现明确生成痕迹 |
| AI 生成 CI/CD 配置 | 无法判断 | 未发现 CI/CD 配置文件 |
| AI 生成 Docker/IaC 配置 | 未发现 | 未发现相关配置 |
| 开发者手动引入 | 无法排除 | 需要结合提交记录确认 |
| 未知来源 | 是 | 当前缺少依赖引入来源记录 |
```

推荐结论写法：

```text
当前证据不能直接证明风险一定由 AI 生成，但可以确认该风险符合 Vibe Coding 场景下常见的进入方式：
开发者快速接受依赖或脚本后，风险进入依赖清单，并可能继续影响构建链和运行期。
```

这句话很重要，因为它既贴合主题，又不会过度宣称。

## 把扫描结果按攻击阶段分类

报告不要只写“发现几个漏洞”，应该按供应链攻击阶段组织。

建议使用下面结构：

```text
## 六、攻击阶段研判

### 1. 入口风险

高风险依赖进入依赖清单，首要对象为 axios。

证据：
- package.json / package-lock.json 中存在相关依赖记录。
- GNN 风险分较高。
- GraphRAG 将该依赖召回为攻击路径入口节点。

### 2. 执行风险

需要检查依赖安装脚本、postinstall、构建脚本和远程命令执行行为。

证据：
- 当前依赖清单已识别。
- 若存在 postinstall、curl | bash、远程脚本下载，需要标记为执行风险。

### 3. 构建风险

当前未发现 CI/CD 配置，因此构建链证据不足。

影响：
- 无法判断 runner 是否可信。
- 无法判断 workflow 权限是否过大。
- 无法判断 Action 是否固定版本。

### 4. 产物风险

压缩包或二进制文件需要进入产物可信校验。

检查项：
- artifact SHA256 是否匹配。
- provenance 是否来自预期 workflow。
- builder 是否可信。
- commit/ref 是否匹配。
- 签名是否存在。

### 5. 运行期风险

运行期证据显示 app -> 45.83.64.12 的外联行为。

影响：
- 可能存在异常通信。
- 需要确认是否涉及用户数据、Token 或业务接口。

### 6. 影响风险

当前需要继续确认：
- 是否影响用户数据；
- 是否影响构建凭据；
- 是否影响发布渠道；
- 是否影响下游系统；
- 是否需要回滚版本；
- 是否需要吊销 Token。
```

## 增加“先处理什么”的处置清单

报告最后要给用户清晰动作，不要只给抽象建议。

推荐格式：

```text
## 七、处置建议

### 立即处理

1. 暂停使用高风险依赖对应版本。
2. 在干净环境中重新安装依赖并重新构建。
3. 校验产物哈希、签名和 provenance。
4. 排查 app -> 45.83.64.12 的运行期外联。

### 短期修复

1. 固定依赖版本，开启 lockfile 审查。
2. 引入 Git Hook，在提交前阻断硬编码密钥和可疑依赖。
3. 引入 CI Gate，在 PR 阶段阻断高危依赖、危险 workflow 和过宽权限。
4. 引入 Release Gate，在发布前校验 artifact、attestation 和 builder。

### 后续增强

1. 保存 AI 生成代码、AI 推荐依赖、AI 生成 workflow 的来源标记。
2. 为每次构建生成 SBOM 和 provenance。
3. 将运行期日志持续接入知识图谱，形成持续溯源能力。
```

## 建议的完整报告目录

最终导出的 Markdown 报告建议采用这个目录：

```text
# SupplyGuard KG 供应链攻击溯源报告

## 0. 研判助手摘要
一句话说明最高风险链路、首要风险对象、运行期证据和处置建议。

## 1. 扫描范围
说明项目、文件总数、可扫描文件、依赖清单、CI/CD 配置、主要语言。

## 2. 工程证据清单
列出实际扫描到的 package.json、lockfile、requirements.txt、workflow、Dockerfile、日志、产物等。

## 3. 证据完整性提醒
说明哪些材料缺失，例如 CI/CD 配置缺失、产物未上传、日志缺失、二进制被跳过。

## 4. Vibe Coding 风险归因
说明风险是否可能来自 AI 推荐依赖、AI 生成代码、AI 生成 CI/CD 配置或未知来源。

## 5. GraphRAG + GNN 智能证据
展示图谱节点、图谱边、攻击路径、GNN 覆盖、模型类型和高风险依赖排序。

## 6. 攻击阶段研判
按入口、执行、构建、产物、运行、影响六个阶段解释风险。

## 7. 攻击路径还原
用自然语言和路径图说明风险如何从依赖进入构建链，并在产物或运行期得到印证。

## 8. 用户影响评估
回答是否影响用户数据、构建凭据、发布渠道、下游系统。

## 9. 处置建议
给出立即处理、短期修复和后续增强。

## 10. 附录
保留原始 findings、证据节点、图谱边、扫描配置和策略文件。
```

## 前端报告页可以怎么改

建议把当前报告页改成从上到下五层：

```text
第一层：研判助手摘要
- 一句话结论
- 首要风险对象
- 最高风险路径
- 立即处置建议

第二层：扫描范围和证据完整性
- 文件总数
- 可扫描文件
- 依赖清单数量
- CI/CD 配置数量
- 跳过的二进制 / 压缩包
- 缺失证据提醒

第三层：GraphRAG + GNN 智能证据
- 图谱节点
- 图谱边
- 攻击路径数量
- GNN 覆盖
- 高风险依赖排序

第四层：攻击路径流程
- 入口风险
- 执行风险
- 构建风险
- 产物风险
- 运行风险
- 影响风险

第五层：处置和导出
- 立即处理
- 门禁建议
- 需要补充的证据
- 导出 Markdown / HTML / 证据包
```

## 报告中需要避免的表达

不要写：

```text
GNN 90%，说明该依赖就是恶意包。
```

建议写：

```text
GNN 90% 表示该依赖在当前供应链图谱中的风险排序较高，需要结合漏洞、脚本、版本、构建链和运行期证据综合判断。
```

不要写：

```text
本次攻击一定由 AI 生成。
```

建议写：

```text
当前证据不能证明风险一定由 AI 生成，但该风险符合 Vibe Coding 场景中“AI 推荐 / 用户接受 / 快速进入工程流程”的典型风险进入方式。
```

不要写：

```text
未发现 CI/CD，所以没有构建风险。
```

建议写：

```text
未发现 CI/CD 配置，因此构建链风险无法完整判断，需要补充 workflow、构建日志或 provenance 材料。
```

## 可以直接放进报告的摘要模板

```text
本次扫描显示，项目存在一条需要优先处理的供应链风险路径：
高风险依赖进入依赖清单，并在知识图谱中与运行期异常外联形成关联。

当前首要风险对象为 {package_name}，风险分 {risk_score}；
运行期证据显示 {process_name} -> {remote_ip}。

GraphRAG 从知识图谱中召回 {graph_nodes} 个节点、{graph_edges} 条边和 {attack_paths} 条攻击路径；
GNN 已覆盖 {gnn_covered}/{gnn_total} 个依赖节点，用于辅助识别高风险依赖和相似恶意包。

建议立即隔离高风险依赖，使用干净 runner 重新构建，校验 artifact 哈希、签名和 provenance，并排查运行期外联。
如果确认产物或运行环境已经受影响，应进一步回滚版本、吊销相关 Token，并检查用户数据和下游系统影响范围。
```

## 与比赛叙事的关系

这次改造能让报告更贴近你的参赛故事：

```text
Vibe Coding 不是单独的漏洞类型，而是风险进入软件工程流程的新方式。
SupplyGuard KG 的价值，是把 AI 推荐依赖、AI 生成脚本、CI/CD 配置、产物可信和运行期日志串成一条可解释的供应链攻击链。
```

这样讲的好处是：

- 不会偏离 APT 供应链攻击检测与溯源主题。
- 能突出 Vibe Coding 场景的新风险。
- 能说明你的系统不是只扫漏洞，而是在还原攻击路径。
- 能把 GNN、GraphRAG、知识图谱和 Agent 后端都自然放进报告能力里。
